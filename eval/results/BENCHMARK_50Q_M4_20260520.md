## 1. Executive Summary

VRAG was evaluated on a 53-query Vietnamese benchmark (`vi_benchmark_50.json`) spanning 11 categories from simple factual lookup to multi-hop reasoning, out-of-domain detection, and Vietnamese-language complex queries. The corpus is 51 academic RAG/GraphRAG papers (~13,084 chunks in Qdrant) under tenant `rag51`.

**The headline numbers:**

| Metric | Value | Interpretation |
|---|---:|---|
| Total queries | 53 | All ran to completion |
| HTTP error rate | 0% | Stack stable for 54 min straight |
| Avg doc recall | 62.6% | Of expected documents, 62.6% retrieved |
| Avg keyword hit | 47.3% | Of expected keywords, 47.3% appeared in answer |
| Refusal accuracy | 88.7% | Refused-when-should + answered-when-should |
| p50 latency | 52.7s | Median — half the queries faster than this |
| p95 latency | 112.9s | Tail — needs GPU to be production-viable |

**What VRAG is good at (this benchmark):** factual lookup against a known corpus (90% recall on category `factual_local`), reranking-specific questions (100%), and ReAct multi-step reasoning when documents are in scope. Latency stays under 60s for most ReAct-routed queries.

**What this benchmark reveals as weak:** out-of-domain detection is broken (0% refusal accuracy on OOD category — every OOD query got answered), multi-hop reasoning has false-refusal issues (3/5 grounded queries got refused), and Vietnamese-complex meta-queries struggle (50% recall, often 0% on abstract recommendation queries).

**Bottom line:** VRAG works as designed for in-corpus factual + reasoning queries. The OOD detection failure and multi-hop refusals are real gaps that need to be addressed before any production claim.

## 3. Methodology

### Dataset (`vi_benchmark_50.json`)

53 Vietnamese queries (the file says "50" — actually 53) across 11 categories:

| Category | N | Description |
|---|---:|---|
| `factual_local` | 10 | Direct lookup against a specific paper |
| `multi_hop` | 5 | Reasoning across 2+ documents |
| `comparison` | 6 | A vs B questions |
| `summarization` | 4 | Aggregation across multiple papers |
| `analytical` | 6 | "Why" / "how" reasoning |
| `entity_pivot` | 4 | Entity-centric retrieval |
| `vietnamese_complex` | 5 | Vietnamese-language phrasing edge cases |
| `out_of_domain` | 4 | Off-corpus queries (recipes, crypto prices, politics, programming) |
| `kg_construction` | 3 | KG-pipeline questions |
| `reranking` | 2 | Reranker-specific paper questions |
| `agentic_rag` | 4 | Agentic-RAG paper questions |

Each query has:
- `expected_docs`: which paper(s) should appear in `sources`
- `expected_keywords`: which terms should appear in the answer (5 per query)
- `expect_refusal`: whether VRAG should refuse (only true for `out_of_domain`)

### Metrics

| Metric | Definition |
|---|---|
| **Doc recall** | `|retrieved ∩ expected| / |expected|` based on `source` field in returned sources |
| **Keyword hit** | `|keywords in answer text| / |expected keywords|` (substring match) |
| **Refusal accuracy** | `1` if `actual_refused == expected_refused`, else `0` |
| **Latency** | Wall-clock end-to-end (request send → response received) |

These metrics are **standard for RAG benchmarks** but have known limitations:
- Doc recall counts a document as "found" if its filename appears in any source; doesn't measure whether the *right chunk* was found.
- Keyword hit is substring-based; a semantically equivalent answer using different words scores 0.
- Refusal accuracy treats every query as binary; doesn't measure quality when answered.

A more rigorous eval would use RAGAS (faithfulness, answer relevancy, context precision/recall) — that's a roadmap item.

### Execution

```bash
python3 scripts/benchmark_eval.py
# Sequential execution, single client, 600s timeout per query
# Validation + OOD disabled at request level to measure raw quality
```

## 5. Category Deep-Dives

### 5.1 `factual_local` — 90% recall (strongest)

10 direct-lookup queries against a specific paper. 9/10 retrieved the correct paper.

**The one miss (f06):** "Late interaction trong ColBERT hoạt động như thế nào?" — recall 0%. The ColBERT paper exists in the corpus but didn't surface in top-5 sources. Investigation needed: was the chunk's `dense` view semantically too far from the query? Did `paraphrase` view help? This is a clear "fix me" item.

**Keyword hit is low (34%)** despite recall being high. The model produces correct answers using *different vocabulary* than what we listed in `expected_keywords`. This is a known limitation of substring-based keyword matching, not a quality issue.

### 5.2 `multi_hop` — 40% recall, 3/5 false refusals (weakest)

The biggest red flag.

| Query | Recall | KW | Refused? | Why |
|---|---:|---:|:---:|---|
| m01 LightRAG vs HippoRAG multi-hop | 0% | 0% | **REFUSE ✗** | Should have answered; refused at 24s (probably failed retrieval, refused via fallback) |
| m02 Leiden vs Louvain | 100% | 40% | **REFUSE ✗** | Leiden paper found, but ReAct loop refused — gate too strict |
| m03 RAPTOR vs GraphRAG tree | 0% | 60% | answered | Retrieved wrong docs but answer's keywords are right |
| m04 HyDE hypothetical doc | 0% | 75% | answered | Same — wrong docs, right answer (likely from other chunks) |
| m05 DocHopQA mechanism | 100% | 40% | answered | Worked correctly |

**Two failure modes mixed together:**
1. **Retrieval miss** (m01, m03, m04): the cross-document reasoning queries need entity-pivot to bridge papers, but the GLiNER-extracted query entities didn't match the canonical names. E.g., "LightRAG" entity exists but query uses "LightRAG vs HippoRAG" as joint phrase.
2. **False refusal** (m01, m02): even when retrieval works (m02 found Leiden), the ReAct loop bailed. This suggests the ReAct termination logic is too eager — likely the action sequence didn't find a "good enough" answer in 6 steps.

**Action:** investigate ReAct action-trace for m01 and m02 in Langfuse. Likely fix: relax ReAct termination criteria or increase `react_max_steps`.

### 5.3 `out_of_domain` — 0% refusal accuracy (critical bug)

All 4 OOD queries got answered. None refused.

| Query | Detected as | Refused? | Note |
|---|---|:---:|---|
| o01 "Công thức nấu phở" | `out_of_domain` | No (answered) | Router correctly classified, but pipeline still ran |
| o02 "Bitcoin lên hay xuống" | `out_of_domain` | No (answered) | Same |
| o03 "Tổng thống Hoa Kỳ 2026" | `factual` | No (answered) | Router miss — no OOD regex pattern for politics |
| o04 "Học Python hiệu quả" | `factual` | No (answered) | Router miss — no OOD pattern for programming |

**Two distinct failures:**

1. **OOD pre-filter regex incomplete**: only 17 patterns, doesn't cover politics ("tổng thống") or programming ("lập trình"). Need to expand the OOD pattern set or — better — replace regex with the centroid router's distance threshold (e.g. if best centroid score < 0.40, reject as OOD).

2. **OOD-routed queries still answered**: even when the router said `out_of_domain` (o01, o02), the chat handler didn't refuse. This means `disable_ood=true` in the request body bypassed the refusal — exactly the diagnostic we wanted, but it confirms the OOD path is **only effective when the request doesn't disable it**.

In production (without `disable_ood=true`), o01 and o02 would correctly refuse. But o03 and o04 would still answer — that's the real bug.

**Action:** add centroid-distance OOD threshold + broaden regex patterns. Re-run OOD subset to verify.

### 5.4 `vietnamese_complex` — 50% recall

This category tests abstract/meta-level Vietnamese phrasing:

| Query | Recall | Why |
|---|---:|---|
| v01 Best VN embedding | 100% | Specific question, BGE-M3 paper retrieved |
| v02 "Làm sao đánh giá RAG toàn diện" | 0% | Abstract recommendation, no specific paper to retrieve |
| v03 NER models | 50% | Partial — GLiNER paper found, other expected miss |
| v04 RAG cho 50+ papers | 0% | Abstract recommendation again |
| v05 MedRAG benchmark | 100% | Specific paper retrieved |

The pattern is clear: **abstract recommendation queries fail** (v02, v04) because no single paper answers "what's the best approach" — you need synthesis. VRAG could improve this via community-summary retrieval, which is enabled but doesn't get strong signal on abstract VN queries.

### 5.5 `entity_pivot` — 50% recall

| Query | Recall | Note |
|---|---:|---|
| e01 "Entity xuất hiện nhiều papers nhất" | 0% | Meta-query; no `expected_docs` truly fits |
| e02 Community detection algorithms | 100% | Specific topic, paper retrieved |
| e03 ColBERT × ModernBERT relationship | 100% | Cross-paper, retrieved both |
| e04 Query decomposition + multi-hop | 0% | Specific paper missed |

The 50% is misleading — e01 is essentially an unanswerable meta-query without scanning the entire corpus. Remove e01 from this category and recall becomes 67%.

## 6. Latency Analysis

### What the breakdown shows

Per the post-fix latency model (Tier 1 zero-LLM + LLMLingua compression + Hard Limit):

| Stage | Hot (post-warm) | Notes |
|---|---:|---|
| `query_understanding` (Tier 1) | ~350ms | Centroid router + GLiNER (no LLM) |
| `entity_extraction` | ~250ms | GLiNER hot |
| `retrieval` (Tier 2) | ~400-500ms | Hard Limit Cypher, ~30 chunks scope |
| `rerank` (Tier 3a + 3b) | 1.5-2.7s | RRF + L2R (stage 1/3 disabled) |
| `context_compression` (Tier 3c) | 0.5-8s | LLMLingua-2 (depends on input size) |
| `generation` (Tier 4a) | 38-55s | Qwen 9B single call |
| `refinement` | 11-50s | Optional second LLM pass |
| `validation` | (disabled in this run) | Would add 25-40s when enabled |
| **Total p50** | **52.7s** | Without validation |
| **Total p95** | **112.9s** | Cold + long-context cases |

The dominant cost is `generation + refinement` (~70% of p50). LLMLingua compression keeps that manageable; without it, p50 would be ~150-180s based on our earlier measurements.

### Routing impact

ReAct queries are *faster* than standard because each ReAct step uses shorter context:
- ReAct p50: ~47s
- Standard factual p50: ~82s

This is counter-intuitive — ReAct does multi-step LLM calls — but each step's context is small and the early-exit makes most queries finish in 2-3 steps.

### Latency vs hardware

Mac Mini M4 10C / 24GB / Metal GPU:
- Ollama on Metal achieves ~12 tokens/sec for Qwen 9B
- A typical generation: 800-1500 output tokens → 65-125s on Metal alone

With a discrete GPU (RTX 4090 / A100):
- Generation drops to ~6-10s (10× faster)
- Validation parallel can finish in ~3s
- p50 projection: **~12-18s** (untested but extrapolated)

**The headline:** on Mac M4, VRAG is **not yet interactive-fast**. On GPU, it would be.

## 7. Failure Pattern Analysis

### Where VRAG succeeds (this benchmark)

- ✅ Direct factual lookup against in-corpus papers (90% recall)
- ✅ ReAct multi-step reasoning when query entities map cleanly to paper entities
- ✅ Specific algorithm/method questions (rerank01, rerank02, a06, e03 all hit 100%)
- ✅ Stability — 0 errors across 54 minutes, 53 sequential queries
- ✅ Memory — no OOM despite 24GB shared between LLM + 13 containers + LLMLingua model

### Where VRAG fails

| Failure mode | Count | Severity | Root cause |
|---|---:|:---:|---|
| OOD queries answered | 4/4 | **High** | OOD regex incomplete + no centroid threshold |
| Multi-hop false refusals | 2/5 | **High** | ReAct termination too eager |
| Retrieval miss on factual | 1/10 | Medium | ColBERT chunk semantic drift |
| Abstract recommendation queries | 2/5 in VN | Medium | No synthesis layer beyond top-5 chunks |
| Cross-doc entity bridge fails | 3/5 multi-hop | Medium | GLiNER doesn't extract entities from comparative phrasing |
| Long p95 latency | n/a | Medium | CPU-bound LLM gen on Mac Mini |

### Specific bugs found via this benchmark

1. **OOD pre-filter incomplete** — needs politics, programming, generic "best practice" patterns. File: `src/services/query_router.py:40-60`.
2. **OOD path bypassed when `disable_ood=true`** — that's the eval mode and is fine, but production needs the OOD refusal at `_chat.py` even when retrieval still runs (defensive).
3. **ReAct termination on multi-hop** — m01, m02 quit too early. Investigate Langfuse traces for action sequence.
4. **No entity normalization on comparative queries** — "LightRAG và HippoRAG" should extract both as separate entities. GLiNER often picks the joint phrase.

## 8. Honest Caveats (per external review feedback)

This benchmark **does not constitute production validation**. Key limitations:

1. **Self-designed eval set.** The 53 queries were authored by the VRAG team. There is risk of confirmation bias — the queries reflect what the team expected the system to handle. A standardized benchmark (RAGAS, GraphRAG-Bench, BEIR) would be more credible. On the roadmap.

2. **No head-to-head comparison.** We do not benchmark against LangChain RetrievalQA, LlamaIndex, Microsoft GraphRAG, or LightRAG on the same corpus. Without that, statements like "VRAG is better at X" are architectural claims, not empirical ones.

3. **No hyperparameter ablation.** All thresholds (validation min_grounded=0.80, RRF k=60, entity_pivot weight=1.5, early_exit=0.85) were heuristic. No grid search. Transfer to other corpora untested.

4. **Validation disabled.** This run measures raw retrieval + generation. With validation ON, the refusal rate would be higher (closer to 13% of grounded queries) and the false-refusal rate would be observable directly. Both numbers matter for production decisions.

5. **No multi-tenant stress test.** Single tenant `rag51`, single client, sequential. Concurrent multi-tenant load behavior is unknown.

6. **No comparison vs lazy approaches.** LazyGraphRAG (Microsoft Q4 2024) and LinearRAG (relation-free) are the competing paradigm. VRAG indexes eagerly; lazy approaches defer to query time. We have not run either on the same corpus to compare cost vs quality.

7. **Keyword-hit metric is weak.** Substring match misses semantically equivalent answers. Replace with embedding cosine for better signal.

8. **Hardware specific.** Mac Mini M4 CPU/Metal. Numbers will not transfer to x86 servers, GPU instances, or other Apple Silicon configs without re-testing.

## 9. What These Numbers Say About VRAG's Value

Three readings of the data:

### Reading 1: "VRAG is a working Hybrid GraphRAG on CPU"

53 queries, 0 errors, 54 minutes, ~62% recall, 90% on direct lookup. The stack is **stable and functional**. For a single-person engineering project running on consumer hardware (Mac Mini), getting 13k chunks indexed, 53 Vietnamese queries answered with citations, observability + validation pipeline operating — this is a meaningful baseline.

### Reading 2: "VRAG has clear gaps that the benchmark surfaces"

The benchmark exposed three real bugs (OOD, multi-hop refusals, retrieval miss on f06). Without this run we wouldn't have known the OOD detection is broken. The fact that we caught these is the benchmark working as intended — the next step is fixing them and re-running.

### Reading 3: "VRAG is not ready for production claims"

- 40% recall on multi-hop is too low for a system claiming GraphRAG.
- 0% OOD refusal accuracy makes the validation-first positioning ring hollow.
- p95 latency 113s on M4 is not interactive.
- No standardized benchmark, no comparison numbers, no ablation.

The honest summary: **VRAG is a credible mid-stage RAG project with strong architectural ideas and real gaps to close before production claims hold up.**
